================================================================================
1. RESUMEN EJECUTIVO
================================================================================

Se realizaron pruebas automatizadas sobre los servicios REST de registro
(signup) y autenticación (login) de demoblaze.com utilizando Cypress.io.
Se cubrieron 4 casos de prueba principales más verificaciones adicionales,
logrando una cobertura completa de los escenarios de negocio definidos.

================================================================================
2. HALLAZGOS TÉCNICOS
================================================================================

2.1 COMPORTAMIENTO DE LA API DE SIGNUP (/signup)
------------------------------------------------
  • Método: POST
  • Content-Type: application/json
  • Payload: {"username": "string", "password": "string"}

  Hallazgo #1: RESPUESTA DE ÉXITO ES UN STRING VACÍO
    Cuando el registro es exitoso, la API devuelve un body completamente
    vacío (string ""). Esto es poco convencional; la mayoría de APIs
    devuelven un objeto JSON con datos del usuario creado o al menos un
    mensaje de confirmación.

  Hallazgo #2: VALIDACIÓN DE USUARIO DUPLICADO
    La API valida correctamente si un usuario ya existe y devuelve:
    {"errorMessage": "This user already exist."}

  Hallazgo #3: NO VALIDACIÓN DE FORTALEZA DE PASSWORD
    La API acepta cualquier password sin validar longitud mínima,
    complejidad o caracteres especiales. Se podría registrar un usuario
    con password de un solo carácter.

2.2 COMPORTAMIENTO DE LA API DE LOGIN (/login)
----------------------------------------------
  • Método: POST
  • Content-Type: application/json
  • Payload: {"username": "string", "password": "string"}

  Hallazgo #4: AUTENTICACIÓN CON TOKEN
    En login exitoso, la API devuelve:
    {"Auth_token": "valor_del_token"}
    El token parece ser un string alfanumérico que permite la autenticación
    en subsiguientes peticiones.

  Hallazgo #5: MENSAJES DE ERROR ESPECÍFICOS
    La API distingue entre dos tipos de errores de login:
    - Password incorrecto (usuario existe): {"errorMessage": "Wrong password."}
    - Usuario inexistente: {"errorMessage": "User does not exist."}
    Esto puede ser una vulnerabilidad de seguridad (user enumeration)
    ya que permite a un atacante saber qué usuarios están registrados.

2.3 COMPORTAMIENTO GENERAL DE STATUS CODES
------------------------------------------
  Hallazgo #6: USO INADECUADO DE STATUS CODES HTTP
    La API devuelve status 200 OK para TODAS las respuestas, incluso
    cuando ocurren errores de negocio. Los códigos HTTP apropiados
    deberían ser:
    - 201 Created para signup exitoso
    - 409 Conflict para usuario duplicado
    - 401 Unauthorized para credenciales incorrectas
    - 404 Not Found para usuario inexistente
    El uso exclusivo de 200 dificulta la detección automática de errores
    y no sigue las mejores prácticas REST.

================================================================================
3. CONCLUSIONES
================================================================================

4.1 SOBRE LA CALIDAD DE LA API
------------------------------
  ✓ La API es funcional y responde a todas las operaciones básicas.
  ✓ Los endpoints están disponibles y responden en tiempos razonables.
  ✗ El diseño de respuestas no sigue estándares REST (status codes).
  ✗ La respuesta de éxito en signup es un string vacío, poco informativo.
  ✗ Existe una vulnerabilidad de enumeración de usuarios.

4.2 SOBRE CYPRESS.IO COMO HERRAMIENTA DE PRUEBAS API
-----------------------------------------------------
  ✓ Cypress permite probar APIs REST de manera efectiva con cy.request().
  ✓ Los comandos personalizados facilitan la reutilización de código.
  ✓ Los logs detallados permiten identificar entradas y salidas claramente.
  ✓ La generación automática de videos y screenshots ayuda en el análisis.

4.3 RECOMENDACIONES
-------------------
  1. Implementar códigos de estado HTTP apropiados (201, 401, 409, 404).
  2. Devolver información del usuario creado en la respuesta de signup.
  3. Implementar validación de fortaleza de contraseñas.
  4. Unificar el mensaje de error de login para evitar enumeración de usuarios.
  5. Considerar agregar rate limiting para prevenir ataques de fuerza bruta.
